## Tokenizer research

- Installed versions inspected: `sentence-transformers` 5.6.0, `transformers` 5.13.1,
  `tokenizers` 0.22.2, and `huggingface-hub` 1.23.0.
- Context7 documentation confirmed that fast tokenizer offsets refer to original input character
  spans and that `add_special_tokens=False` is required for the Phase 3 count contract.
- `SentenceTransformer` construction loads model modules/weights and can download uncached assets;
  merely using its tokenizer property therefore requires a managed model instance.
- An offline `local_files_only=True` probe found no cached tokenizer for the configured default
  model and performed no download.
- Phase 4 should adapt the tokenizer from one cached Sentence Transformer lifecycle rather than
  constructing a second model or tokenizer lifecycle.

## Security review result

- Codex Security preflight ran once and returned `ready` with exit code 0.
- The delegated review completed without a content-policy blocker and found no validated high- or
  critical-severity issue and no reportable finding.
- Manual review confirmed no filesystem/network/model loading, source logging, Python `hash()`,
  Graphify influence, unrelated-symbol overlap, or unstable hash/ID encoding.
- One low-severity local-availability hardening suggestion remains: mandatory signature context is
  retokenized for split candidates. With a 10,023-character tokenizer-cheap signature and a
  13,604-character synthetic source, the bounded review measured 2,617,564 processed characters,
  or 192.4x source length, across 41 chunks and 248 count calls.
- No Phase 3 change was made for that suggestion because arbitrary character caps or signature
  rewriting would conflict with the exact signature-context contract. Revisit it at the Phase 4
  tokenizer/indexer boundary alongside configured file-size enforcement and production tokenizer
  lifecycle.
